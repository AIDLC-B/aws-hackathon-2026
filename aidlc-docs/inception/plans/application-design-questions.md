## 質問 1: バックエンドのアーキテクチャスタイル
バックエンドのアーキテクチャはどのスタイルにしますか？

A) ドメインごとに独立したLambda関数群（各ドメインが独立したLambdaセット）
B) 単一のLambda関数でルーティング（モノリシックLambda）
C) ドメインごとに独立したAPI Gateway + Lambda（マイクロサービス的）
D) その他（[Answer]: タグの後に記述してください）

[Answer]: C

## 質問 3: データベース設計方針
DynamoDBのテーブル設計はどうしますか？

A) シングルテーブル設計（1テーブルに全エンティティ）
B) エンティティごとにテーブルを分ける（料理テーブル・献立テーブルなど）
C) その他（[Answer]: タグの後に記述してください）

[Answer]: C（今の想定はCognitoによるログインによってユーザーごとIDを特定し、そのUIDを用いてドメインごとにリクエストする。各ドメインごとのドメインアクセス経路は以下の通り。
・フロントエンド→献立提案→料理管理
・フロントエンド→献立ガチャ→料理管理
・フロントエンド→AIキャラクター
・フロントエンド→料理管理
※ユーザIDは必ずリクエストに含まれる
なお、献立提案の時に確定済みの献立は除外する。そのために確定済みの料理は献立提案ドメインに属する。）

## 質問 4: 画像ストレージとAI認識
料理写真のアップロードとAI認識のフローはどうしますか？

A) フロントエンド → S3直接アップロード（Presigned URL）→ Lambda → Bedrock認識
B) フロントエンド → Lambda → S3アップロード → Bedrock認識
C) その他（[Answer]: タグの後に記述してください）

[Answer]: B（フロントエンド側のハンドリングを考慮して同期型を選択する。）

## 質問 5: キャラクタードメインのBedrock呼び出し
キャラクターの一言生成はどのタイミングでBedrockを呼び出しますか？

A) リアルタイム（ユーザーのアクション時に毎回Bedrockを呼び出す）
B) 事前生成（起動時やバックグラウンドでセリフをキャッシュしておく）
C) ハイブリッド（一部はキャッシュ、SSR演出など特別なケースはリアルタイム）
D) その他（[Answer]: タグの後に記述してください）

[Answer]: C（応答速度を必要な場合はキャッシュ、それ以外はリアルタイムで呼び出す）
