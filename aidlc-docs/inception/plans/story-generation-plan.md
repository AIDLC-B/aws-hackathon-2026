# ストーリー生成計画（確定版）

## 計画サマリー

| 項目 | 決定内容 |
|---|---|
| **ペルソナ** | 1ペルソナ（主婦・主夫の代表的な1人） |
| **分解アプローチ** | ドメインベース（フロントエンドを結合レイヤーとして、バックエンドでドメイン分割） |
| **Acceptance Criteria形式** | Given / When / Then（BDD形式） |
| **優先度付け** | MoSCoW法（Must / Should / Could / Won't） |

## ドメイン構成（4ドメイン）

| ドメイン | 対応FR | 説明 |
|---|---|---|
| 料理管理ドメイン | FR-01 | 料理登録・レパートリー管理・レアリティ設定・検索APIの提供 |
| 献立提案ドメイン | FR-02, FR-04 | フィルタリングロジック・ワンボタン提案・ガチャ誘導判定・ストリーク・週次レポート |
| ガチャドメイン | FR-05 | ガチャ演出・レアリティ抽選・リセマラ管理 |
| キャラクタードメイン | FR-03, FR-06 | 一言生成・キャラクター/トーン選択（内部隠蔽）・Bedrock連携・プレミアムフラグ参照 |

## ドメイン間結合方針（Option A: 料理管理がAPIを提供）

```
[フロントエンド（React）] <- 結合レイヤー
        |
        +-> 料理管理API     : 料理の登録・更新・削除・検索APIを提供
        |        ^
        +-> 献立提案API     : 料理管理APIを呼び出してリストを取得し、
        |                     フィルタリング・3品選択を実施
        |
        +-> キャラクターAPI : タイミング・状況のみを受け取る
                              キャラクター選択・トーン選択は内部に隠蔽
```

**責務の分担**

| 責務 | 担当ドメイン |
|---|---|
| 料理の登録・更新・削除 | 料理管理 |
| 料理の検索APIの提供（条件付きクエリ） | 料理管理 |
| フィルタリングロジック（質問形式→10件以内絞り込み） | 献立提案 |
| 3品のランダム選択 | 献立提案 |
| レパートリー件数確認（3品未満→ガチャ誘導判定） | 献立提案（料理管理APIを呼ぶ） |
| キャラクター選択・トーン選択ロジック | キャラクター（内部に隠蔽） |
| Bedrock連携・一言テキスト生成 | キャラクター（内部に隠蔽） |
| プレミアムフラグ参照（推しキャラ固定） | キャラクター（内部に隠蔽） |

**キャラクタードメインのインターフェース設計（確定）**

呼び出し元（献立提案・ガチャ）は「タイミング・状況」のみを渡す。
キャラクター選択・トーン選択・Bedrock呼び出しはすべてキャラクタードメイン内部に隠蔽する。

呼び出し側が渡す情報（状況コンテキスト）の例：

| 状況 | 渡す情報 | 内部で決定すること |
|---|---|---|
| 献立確定時 | `trigger: "meal_decided", from: "suggestion"` | キャラクター・トーン・セリフ |
| 料理登録時 | `trigger: "recipe_registered", from: "registration"` | キャラクター・トーン・セリフ |
| ガチャリセマラ5回目 | `trigger: "gacha_reroll_limit", from: "gacha"` | メシストフェレス固定・セリフ |
| 献立提案時（通常） | `trigger: "meal_suggested", from: "suggestion"` | キャラクター・トーン（ランダム）・セリフ |
| ガチャ結果確定時 | `trigger: "gacha_decided", from: "gacha"` | キャラクター・トーン・セリフ |

**渡す情報の構造（概念）**:
- `trigger`: 何が起きたか（イベント種別）
- `from`: どの画面・フローから遷移してきたか（画面遷移履歴・コンテキスト）
- レアリティ・キャラクター・トーンは呼び出し元から渡さない（キャラクタードメイン内部で判断）

呼び出し元はレスポンスとして「キャラクター名・セリフ・表示用メタ情報」を受け取るだけでよい。

## 料理登録の入力項目（確定）

| 項目 | 必須/任意 | 備考 |
|---|---|---|
| 料理名 | 必須（写真なし時）/ 自動入力（写真あり時） | AI認識後に修正可 |
| 写真 | 任意 | あればAIが料理名を自動入力 |
| 難易度 | 必須 | かんたん / ふつう / がんばる |
| 所要時間 | 必須 | 分単位で入力 |
| レアリティ | 必須 | N / R / SR / SSR をユーザーが手動選択 |

## 献立提案のフィルタリングロジック（確定）

1. レパートリーが3品未満 → ガチャへ誘導
2. レパートリーが多い場合 → 質問形式でフィルタリング → 10件以内に絞り込み → ランダム3品提案
3. ユーザーが3品から1品を選択して確定

## ガチャ業務ルール（確定）

| ルール | 内容 |
|---|---|
| レアリティ出現確率 | N: 60% / R: 25% / SR: 12% / SSR: 3%（管理者設定変更可） |
| リセマラ管理 | 5回連続でメシストフェレス登場 → デリバリー誘導 |
