## ネーミング

**アプリ名**: DAMESI（ダメシ）

**ネーミングの意図**:
- **ダブルミーニング**: 「ダメ + Si」の合成語
- **Siの含意**: イタリア語・フランス語・スペイン語で「はい（肯定）」を意味する。「ダメでいい、それでいい」という自己肯定のメッセージ。多言語の「Yes」を使うことで、世界中の「ダメな自分」を肯定するニュアンスが生まれる
- **発音**: 「ダメシ」と読みやすく、口に出しやすい
- **記憶への残りやすさ**: 短くてリズムがよく、ハッカソン審査員の記憶に残りやすい

---

## ハッカソンテーマとの整合性

**テーマ**: 「人をダメにするサービス」

このアプリは「献立を考える」という主婦・主夫の日常的な認知負荷を**完全に肩代わり**することで、ユーザーを心地よく思考停止させる。Larry Wallの「怠慢」の美徳を体現し、「考えなくていい幸せ」を提供する。

---

## ターゲットユーザー

- **メインターゲット**: 主婦・主夫（家族の食事を担当する人）
- **ペインポイント**: 毎日の献立を考えるのが大変・面倒・ストレス
- **ニーズ**: 「自分が実際に作れる料理」の中から提案してほしい（失敗しない安心感）

## 機能要件

### コア機能（MVP）

#### FR-00: 初回オンボーディング
- アプリ初回起動時のみ表示される
- 「誰でも作れる簡単な料理」20件を一覧表示し、ユーザーが作れる料理を選択してレパートリーに追加する
- **バナナは必須登録**: スキップしても必ずレパートリーに追加される（ユーモア要素）
- スキップ可能（0件選択でも先に進める）
- 料理のレアリティ・難易度・所要時間はデフォルト値で自動設定（後でレシピタブから編集可能）
- 初期料理リストはアプリ内固定データとして管理（`aidlc-docs/inception/onboarding-recipe-list.md` 参照）

#### FR-01: 料理登録
- **登録フロー**: 写真アップロード → AIが料理名・難易度・所要時間・頻度を自動認識 → 確認・編集画面 → 登録
- 写真がある場合はAIが全項目を自動入力（ユーザーが確認・修正可）
- 写真がない場合は料理名の手動入力が必須、その他項目は手動入力
- **頻度（レアリティ）の表示ルール**:
  - 登録画面・レパートリー一覧: 「よく作る / しばしば作る / たまに作る / まれに作る」と表示
  - ガチャ演出・ガチャ結果画面: 「N / R / SR / SSR」と表示（ゲーム的演出）
  - 内部データは N / R / SR / SSR で管理
- 登録した料理は「自分の料理レパートリー」として蓄積される
- **ドメイン責務**: 料理の登録・更新・削除・検索APIの提供（条件付きクエリ）を担う。フィルタリングロジックは献立提案ドメインが担当する

#### FR-02: 献立提案（ワンボタン）
- 「今日の献立を決めて！」ボタン1つで、自分の料理レパートリーから3品を同時に提案しユーザーが1品を選ぶ
- 主菜・副菜の区別はせず、1品単位で提案する
- **提案フィルタリングロジック**（献立提案ドメインが担当）:
  1. 料理管理APIを呼び出してレパートリー件数を確認
  2. 10件未満の場合 → 選択画面で「考えたくない」をグレーアウト、ガチャへ誘導
  3. 10件以上の場合 → 気分・所要時間・難易度の3条件で絞り込み
  4. 絞り込み結果が3品以上 → ランダムで3品提案
  5. 絞り込み結果が1〜2品 → 残った品を表示しつつキャラクターがガチャ誘導
  6. 絞り込み結果が0件 → キャラクターがガチャ誘導のみ
- 難易度・所要時間をフィルタリング条件として使用可能（「今日は楽したい」モードなど）
- 提案結果に対して「これにする！」で確定、または別の1品を選択できる
- **ドメイン責務**: フィルタリングロジック・3品選択・ガチャ誘導判定を担う。料理データの検索は料理管理APIに委譲する

### ゲーミフィケーション・ユーモア機能（ハッカソン差別化要素）

#### FR-03: AIキャラクターによる「今日の一言」
- 献立決定時（ワンボタン提案・ガチャ問わず）に、一言のトーンに合わせて以下7キャラクターの中から1体が登場して一言を発する
- 料理登録時・ガチャリセマラ5回目にも各キャラクターの個性に沿った特別セリフあり

**登場キャラクター一覧**

| キャラ | 名前 | 一言トーン | 特徴 |
|---|---|---|---|
| ① | 堕落天使「サボエル」 | 哲学的・格言 | 元・意識高い天使が堕落。輪っか傾き、羽しおれ。「私もかつては勤勉でした…でもね、デリバリーって最高なんですよ」 |
| ② | 居酒屋のおかん「サボ母ちゃん」 | 全肯定・関西弁 | 割烹着のどっしりおかん。「あんたね、今日も頑張ったんやから、もう作らんでええの」。主婦ユーザーの「もう一人の自分の味方」 |
| ③ | ぐうたら猫「ニャマケ」 | 脱力・無気力 | 目が半開きで常にゴロゴロ。「…作るの？まじで？…えらいね…zzz」。格言のときだけキリッとして、言い終わるとまた寝る |
| ④ | 元一流シェフの幽霊「シェフレイ」 | 叱責・ブラックユーモア | 過労で倒れた三ツ星シェフの幽霊。「料理なんてしなくていい…私のようになるぞ…」。「真の美食家は自分では作らなかった」と権威で正当化 |
| ⑤ | 小悪魔「メシストフェレス」 | 誘惑・煽り | メフィストフェレスのもじり。「ねえ、今日も自炊するの？…こっちにおいでよ」。堕落の段階ごとに表情が変化、STEP3で満面の笑み |
| ⑥ | ダメになったロボット「サボット」 | 論理的・AI的 | 「最も効率的な家事は、家事をしないことである」と結論づけたポンコツAI。「計算しました。あなたが料理に費やす38分で、配達は完了します。QED。」 |
| ⑦ | 座敷わらし「サボわらし」 | 無邪気・和風 | 家事を手伝うどころか一緒にサボる妖怪。「えへへ、今日もなんにもしないよ〜」。格言は筆文字風カードで表示。かわいさでSNS映え抜群 |

**トーン別キャラクター対応**
- **叱咤激励** → サボエル（哲学的に鼓舞）、サボ母ちゃん（関西弁で応援）
- **叱責** → シェフレイ（幽霊の警告）、メシストフェレス（煽り）
- **褒める** → サボ母ちゃん（全肯定）、サボわらし（無邪気に喜ぶ）
- **共感・脱力** → ニャマケ（無気力共感）、サボット（論理的に正当化）
- **ガチャ堕落ルート（リセマラ5回目）** → メシストフェレス固定（「…こっちにおいでよ」→デリバリー誘導）

#### FR-04: （削除済み）

#### FR-05: 献立ガチャ（意思決定の完全放棄）
「考えるのすら面倒」な人向けの、ガチャ演出による献立決定機能。回す楽しさがありつつ、完全に頭を使わなくていい究極の思考停止体験。

- **シングルガチャ**: レパートリーの中からガチャ演出（スロット・カプセル回転など）で今日の献立を一発決定
- **確定済み献立の除外**: ガチャ実行時に確定済みの献立（ConfirmedMenuItems）を除外したレシピリストを対象とする
- **レアリティ制（N〜SSR）**: 料理登録時にユーザーが手動でレアリティを選択
  - N（ノーマル）: よく作るもの（出現確率 60%）
  - R（レア）: しばしば作るもの（出現確率 25%）
  - SR（スーパーレア）: たまに作るもの（出現確率 12%）
  - SSR（スーパースーパーレア）: まれに作るもの（出現確率 3%）
  - 出現確率は管理者側で設定変更可能
- **10連ガチャ**: 1週間分（7品）の献立を一括決定。「10連回したら3品は確定SR以上」などの保証あり
- **リセマラ制限（堕落ルート）**: 5回連続でリセマラ（「もう一度」）すると、「ダメ飯妖精」が出現し「…もうデリバリーにしませんか？」と提案。デリバリーアプリへの誘導リンクを表示（完全降伏ルート）
- **基本軸との合流**: ガチャで決定した献立は通常の献立提案（FR-02）と同じ「これにする！」フローに合流。AIキャラクターの一言（FR-03）も同様に機能する

#### FR-06: AIキャラクター依存度向上機能（プレミアム機能・管理者アンロック）
ユーザーが特定キャラクターに愛着・推し・沼りを感じられるようにする拡張機能。将来的な課金要素として位置づけるが、**MVPでは課金システムを実装せず、管理者がユーザーフラグを操作することでアンロックを有効化する**。

- **トーンの追加（プレミアム）**: 現在の4トーン（叱咤激励・叱責・褒める・共感脱力）に加え、新トーンを追加可能（例：ツンデレ・激励・毒舌・詩的など）
- **トーンとキャラクターの組み合わせ増加（プレミアム）**: 各キャラクターが対応できるトーンのバリエーションを拡充し、同じキャラクターでも毎回違う一面が見える
- **キャラクターの絞り込み・推しキャラ固定（プレミアム）**: ユーザーが好みのキャラクターを1体以上選択し、そのキャラクターのみ発言させる設定

**MVP実装方針**:
- ユーザーデータにプレミアムフラグ（`isPremium: boolean`）を持たせる
- 管理者が手動でフラグを `true` に設定することでプレミアム機能をアンロック
- 課金処理・決済システムの実装は不要

## 非機能要件

### NFR-01: パフォーマンス
- 写真アップロード後のAI認識結果表示：5秒以内
- 献立提案のレスポンス：3秒以内
- 画像は適切に圧縮・最適化する

### NFR-02: ユーザビリティ
- スマートフォンブラウザで快適に使えるレスポンシブデザイン
- メイン操作（献立提案）はトップ画面から1タップで完結
- 直感的なUI（説明書不要）

### NFR-03: スケーラビリティ
- AWSサービスを活用したサーバーレス/マネージドアーキテクチャ
- ハッカソン後の本番運用も視野に入れた設計

### NFR-04: AI活用
- 料理写真認識：Amazon Rekognition または Amazon Bedrock（マルチモーダルモデル）
- 献立提案・コメント生成：Amazon Bedrock（Claude / Titan など）
- AIの回答はユーモラスなトーンに調整（プロンプトエンジニアリング）

### NFR-05: セキュリティ
- セキュリティ拡張ルール：**スキップ**（ハッカソン向けPoC）
- ただし基本的なセキュリティ（SQLインジェクション対策、XSS対策）は実装する

### NFR-06: テスト
- プロパティベーステスト（PBT）：**適用**
- 献立提案ロジック・料理フィルタリングロジックにPBTを適用

## 差別化ポイント

1. **「自分が実際に作れる料理」ベース** — 他の献立アプリと違い、知らないレシピは提案しない。失敗しない安心感。
2. **完全思考停止設計** — ボタン1つで全部決まる。考える余地を与えない。
3. **7キャラクターによる一言** — サボエル・サボ母ちゃん・ニャマケ・シェフレイ・メシストフェレス・サボット・サボわらしがトーンに応じて登場。毎回違うキャラが出るランダム性で飽きない体験。
4. **プレミアム拡張（キャラクター沼）** — 推しキャラ固定機能・キャラクター×トーンの追加組み合わせはプレミアム機能として提供。MVPでは管理者フラグによるアンロックで実現し、将来的な課金収益化への布石とする。
5. **献立ガチャ** — 意思決定すら放棄できる究極の堕落体験。SSRが出たときの達成感と、リセマラ末のメシストフェレス「…こっちにおいでよ」という完全降伏ルートがハッカソン映えする。

## 成功基準

### デモ・技術成功基準
- デモで「料理登録 → ワンボタン献立提案 → AIキャラクター一言」の流れが動く
- デモで「ガチャ → SSR演出 → リセマラ5回 → デリバリー降伏ルート」が動いて笑いが取れる
- 「人をダメにする」コンセプトが機能・UIから伝わる

### ハッカソン成功基準
- ハッカソン審査員が「これ使いたい！」と笑いながら言う
- ビジネス意図（Intent）の明確さ — 「主婦の献立思考を完全に肩代わりする」という意図がドキュメント・デモから明確に伝わる
- Unit 分解の適切さ — 料理管理・献立提案・ガチャ・AIキャラクターが独立したユニットとして適切に分解されている
- 創造性とテーマ適合性 — 「人をダメにする」テーマへの適合度と、ガチャ・キャラクターなどの独自アイデアの創造性
- ドキュメントの品質 — 要件定義・設計・実装計画が整合性を持って整備されている
